# TEAM.md — ZCSLIB 多人协作规范

## 核心规则

**每次修改代码后，所有人（含 AI）必须在 `./mem/` 目录写入修改记录。**

不写 → 视为未完成。无例外。

---

## mem/ 目录约定

```
ZCSLIB/
├── mem/
│   ├── 2026-06-24.md      ← 当日修改记录（多人合并写入同一文件）
│   ├── 2026-06-25.md
│   └── ...
```

- **一个自然日一个文件**，文件名 `YYYY-MM-DD.md`
- **所有人写入同一个当日文件**，追加在末尾
- **当天第一个人创建文件**，后续追加

---

## 记录格式

每条记录三要素：

```markdown
### [HH:MM] 作者名 — 一句话概述

**修改内容**：
- 文件1: 改了什么
- 文件2: 改了什么

**原因**：
- 为什么这么改（设计决策/修bug/重构/...）
- 关联的 issue / 讨论（如有）
```

**示例**：

```markdown
### [14:32] 代可行 — Fix PluginLoader 插件计数始终为 0

**修改内容**：
- PluginLoader.java: loadPlugin() 末尾新增 plugins.put()
- ZCSKernel.java: getPluginCount() 改为实时查询

**原因**：
- 实例化后忘记 put 进注册表，导致计数永为 0
- 原缓存字段有陈旧风险，改为实时查询更安全
```

---

## 原则

| 原则 | 说明 |
|---|---|
| **原子** | 一条记录 = 一次修改。不攒到"今天改了很多"写一条 |
| **诚实** | 写实际改了什么，不修饰。改错了也写 |
| **可追溯** | 三个月后回来看，能看懂当时为什么做这个决策 |
| **先写后走** | commit 之前写 mem，不要 commit 后再补 |
| **AI 也写** | AI 通过工具修改代码后，同样写入 mem/ |

---

## 多人写入同一文件注意事项

- 追加，不要覆盖别人的记录
- 如果文件已被别人修改（git pull 后有冲突），按时间戳排序手动合并
- 不要在同一行上编辑，追加即安全

---

## 反模式（禁止）

- ❌ "修了几个 bug" — 必须列具体文件和改了哪行逻辑
- ❌ "重构" — 必须说清楚从什么重构到什么，为什么
- ❌ 只写修改内容不写原因
- ❌ commit message 代替 mem 记录 — mem 是给人看的决策日志，commit 是给 git 看的变更摘要
